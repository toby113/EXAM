## 2. 設計目標與原則
### 2.1 設計目標
1. 以單一 Node.js 服務提供管理端與學生端完整流程。
2. 以明確授權邊界避免學生端取得答案與他人資料。
3. 以 SQLite 強化資料完整性（FK/NOT NULL/CHECK/索引）確保可追溯。
4. 支援 AI 輔助但不綁定單一供應商。

### 2.2 設計原則
- 分層：路由層、資料存取層、LLM 服務層、前端靜態頁。
- 最小揭露：學生 API 僅回傳安全投影欄位。
- 一致性優先：所有提交與明細寫入採交易（transaction）。
- 可演進：schema 變更以 migration 方式累進。

## 3. 架構總覽
### 3.1 邏輯架構
- 表現層：`public/*.html` 與相關前端腳本。
- 應用層：`server.js`（Express 路由與業務邏輯）。
- 資料層：`database.js`（初始化、migration、索引、種子資料）。
- AI 整合層：`llm.js`（OpenAI/Gemini/Claude 統一介面）。
- 檔案層：`uploads/audio`、`uploads/images`。

### 3.2 部署架構
- 單機部署、單一程序。
- 啟動命令：`npm start`。
- 連接埠：`PORT` 環境變數，預設 `3000`。
- 資料庫：`exam.db`（WAL 模式視 SQLite 設定）。

### 3.3 系統邊界與外部介接
| 類別 | 介接對象 | 用途 |
| --- | --- | --- |
| AI Provider | OpenAI | 出題、作文評分、範文生成 |
| AI Provider | Gemini | 出題備援 |
| AI Provider | Claude | 出題備援 |
| 瀏覽器客戶端 | 管理端/學生端 | 題庫、作答、分析、管理 |

## 4. 模組設計
### 4.1 `server.js`
| 模組 | 主要責任 | 關鍵設計點 |
| --- | --- | --- |
| 安全中介層 | `helmet`、`cors`、全域 API limiter | 限制來源與基本安全 header |
| 驗證授權 | `requireAdmin`、`requireStudent`、`requireStudentAccount` | 角色與 ownership 強制檢查 |
| 登入管理 | `/api/login`、`/api/admin/login`、`/api/student/login` | IP + 帳號雙層 rate limiter |
| 題庫管理 | `/api/questions*`（管理端） | 管理端可見答案、含版本與審查流程 |
| 試卷/作答 | `/api/public/exams/*`、`/api/exams/:id/submit` | 作答 session、進度保存、提交交易 |
| 分析與推薦 | `/api/submissions/*`、`/api/recommendations` | 分析報告、能力估算、個人化推薦 |
| 媒體上傳 | `/api/public/audio/upload`、`/api/image/upload` | MIME/大小限制、失敗即刪檔 |

### 4.2 `database.js`
| 類別 | 設計內容 |
| --- | --- |
| 基礎 schema | 建立 `students`、`student_sessions`、`submissions` 等核心表 |
| migration | 補欄位、回填 ownership、重建表以強化 FK/NOT NULL/CHECK |
| 完整性 | `student_account_id` 強制約束、`status` 有效值檢查 |
| 索引 | 查詢/授權/追溯相關索引與唯一索引 |
| 密碼策略 | `scrypt` 雜湊格式，支援相容升級 |

### 4.3 `llm.js`
- 供應商選擇：依設定與可用性選擇 provider。
- 輸入正規化：題型、難度、數量與上下文提示。
- 輸出解析：統一為題目 JSON 物件，回傳驗證錯誤供前端檢視。

### 4.4 `public/` 與 `templates/public/`
- `templates/public/` 作為模板來源，`generate-public.js` 產生 `public/`。
- 主要頁面：登入、考試列表、作答、結果分析、管理後台。

## 5. 安全與授權設計
### 5.1 認證模型
- 管理端：`admin_sessions` + Token Cookie。
- 學生端：`student_sessions` + Token Cookie。
- Session 皆有有效期限與最後活動時間。

### 5.2 授權模型
- 管理 API 全數需 `requireAdmin`。
- 學生敏感 API（作答、推薦、上傳）需 `requireStudent + requireStudentAccount`。
- ownership 判定以 `student_account_id` 為唯一主軸。

### 5.3 防濫用設計
| 控制項 | 設定 |
| --- | --- |
| 全域 API limiter | 每 IP 每分鐘 200 次 |
| 登入 IP limiter | 15 分鐘 30 次（成功不計） |
| 登入帳號 limiter | 同 IP+帳號，15 分鐘 6 次（成功不計） |
| 交卷 limiter | 15 分鐘 10 次 |

### 5.4 學生安全投影
- 設計常數：`STUDENT_SAFE_QUESTION_SELECT`。
- 投影內容不包含 `answer` 欄位。
- 學生取題、推薦題目均套用相同投影，確保一致性。

### 5.5 上傳安全
- 音訊：限制類型 `mp3/wav/ogg/m4a/aac/webm`，大小 50MB。
- 圖片：限制 `jpg/png/gif/webp/svg`，大小 10MB。
- 公開音訊上傳需同時驗證 `submission_id + lookup_token + ownership + status`。

## 6. 資料庫設計
### 6.1 核心實體關係
- `students (1) -> (N) student_sessions`
- `students (1) -> (N) submissions`
- `exams (1) -> (N) submissions`
- `submissions (1) -> (N) answer_details`
- `questions (1) -> (N) answer_details`
- `exams (1) -> (N) exam_questions`

### 6.2 關鍵約束
| 表 | 約束 |
| --- | --- |
| `students` | `username` 唯一；`student_id` 非空唯一索引 |
| `student_sessions` | `student_account_id NOT NULL` + FK `students(id)` |
| `submissions` | `student_account_id NOT NULL` + FK；`status` CHECK |
| `submissions` | `student_session_id` FK `ON DELETE SET NULL` |
| `answer_details` | `submission_id` FK `ON DELETE CASCADE` |

### 6.3 索引策略
- 授權查詢：`idx_submissions_exam_owner`、`idx_student_sessions_account`。
- 結果查詢：`idx_submissions_lookup_token`（唯一）。
- 題庫查詢：`idx_questions_grade_subject_archived`、`idx_questions_review_status`。

## 7. 關鍵流程設計
### 7.1 學生開始作答流程
1. 學生登入取得 session。
2. 呼叫 `/api/public/exams/:id/session`，後端驗證可作答條件與 ownership。
3. 若有 `in_progress` 續作資料則回傳原 session；否則建立新作答紀錄。

### 7.2 作答進度保存與交卷
1. 進度保存：`PUT /api/public/submissions/:id/progress`。
2. 驗證 `lookup_token` 與 ownership。
3. 交卷：`POST /api/exams/:id/submit`，交易化寫入 `submissions` 與 `answer_details`。
4. 同步更新題目統計（答對、答錯、不會）。

### 7.3 口說音訊上傳
1. 學生提交音檔與 `submission_id`、`lookup_token`。
2. 後端驗證作答狀態與 ownership。
3. 驗證失敗則刪除檔案並回錯誤；成功才回傳 `audio_url`。

## 9. 非功能設計
### 9.1 可維護性
- 以 migration 方式遞進 schema，避免直接手工改 DB。
- 將學生端可見欄位集中定義，避免分散式投影遺漏。

### 9.2 可用性與可靠性
- 核心流程設計有錯誤回應與狀態檢查。
- 交卷交易化，降低部分寫入風險。

### 9.3 可觀測性
- 建議將登入失敗率、提交失敗率、429 比率納入日誌監控。
- 發布後 24 小時重點監控授權錯誤與 API 錯誤碼分布。
