## 2. 程式範圍與目標
### 2.1 範圍
本文件描述 `server.js`、`database.js`、`llm.js` 及前端模板生成程式的實作細節、錯誤處理、資料一致性與安全控制。

### 2.2 程式品質目標
1. 關鍵流程有明確授權檢核與資料一致性保護。
2. 所有重大需求可追溯至程式模組與測試項。
3. 缺陷修補可回溯、可驗證、可審計。

## 3. 程式架構與目錄
| 檔案/目錄 | 用途 |
| --- | --- |
| `server.js` | Express API、授權、核心業務邏輯 |
| `database.js` | SQLite 初始化、migration、索引、種子資料 |
| `llm.js` | AI 供應商整合與輸出解析 |
| `generate-public.js` | 由模板產生 `public/` 靜態頁 |
| `public/` | 前端靜態頁面 |
| `templates/public/` | 前端模板來源 |
| `uploads/audio` | 音訊檔案存放 |
| `uploads/images` | 圖片檔案存放 |

## 4. 主要元件詳細設計
### 4.1 `server.js`
#### 4.1.1 啟動與安全中介層
- `helmet`：基礎安全 header（目前 `contentSecurityPolicy: false`）。
- `cors`：依 `ALLOWED_ORIGIN` 控制來源。
- `express.json({ limit: '100kb' })`：限制請求體大小。
- 全域 limiter：`/api/` 每 IP 每分鐘 200 次。

#### 4.1.2 登入與 Session
- 登入端點：`/api/login`、`/api/admin/login`、`/api/student/login`。
- 防暴力機制：
  - `loginIpLimiter`：15 分鐘 30 次。
  - `loginAccountLimiter`：同 IP + 帳號 15 分鐘 6 次。
- 成功登入後建立 session token（hash 存 DB，cookie 回寫客戶端）。

#### 4.1.3 授權中介函式
| 函式 | 用途 |
| --- | --- |
| `requireAdmin` | 僅允許管理端 session |
| `requireStudent` | 僅允許學生 session |
| `requireStudentAccount` | 強制需具有效 `student_account_id` |
| `matchesStudentSubmissionOwner` | 校驗 submission ownership |
| `resolveStudentSubmissionOwner` | 產生 ownership 查詢條件 |

#### 4.1.4 題目與試卷 API
- 管理端題目 CRUD：完整欄位，可讀寫 `answer`。
- 學生端取題：採 `STUDENT_SAFE_QUESTION_SELECT`，不回傳答案。
- 試卷狀態：`draft/active/closed`，並支援時間窗、存取碼、次數限制。

#### 4.1.5 作答流程 API
1. 建立作答 session：`POST /api/public/exams/:id/session`。
2. 進度保存：`PUT /api/public/submissions/:id/progress`。
3. 交卷：`POST /api/exams/:id/submit`。
4. 交卷流程採 transaction 寫入 `submissions` + `answer_details`。

#### 4.1.6 上傳 API
- 管理端上傳：`/api/audio/upload`、`/api/image/upload`。
- 學生端音訊上傳：`/api/public/audio/upload`。
- 驗證失敗情境（submission 不存在、ownership 不符、token 錯誤、狀態不符）皆刪除檔案。

#### 4.1.7 分析與推薦 API
- `GET /api/submissions/:id/analysis`：依 token 查詢分析。
- `GET /api/recommendations`：以學生歷史答題弱點推薦題目。
- 推薦查詢僅依目前登入學生 ownership 取樣。

### 4.2 `database.js`
#### 4.2.1 初始化
- 建立核心表：`subjects`、`questions`、`exams`、`exam_questions`、`submissions`、`answer_details`、`admins`、`students`、`admin_sessions`、`student_sessions`、`question_versions`。

#### 4.2.2 migration 設計要點
1. 增欄位採 `ALTER TABLE ... ADD COLUMN` 兼容舊資料。
2. 先回填 ownership，再進行強約束重建。
3. 重建時使用 transaction 並臨時關閉 FK，以降低中斷風險。
4. 完成後重建索引並恢復 FK。

#### 4.2.3 資料完整性強化
- `student_sessions.student_account_id`：`NOT NULL` + FK。
- `submissions.student_account_id`：`NOT NULL` + FK。
- `submissions.status`：CHECK (`in_progress`,`submitted`)。
- `students.student_id`：非空唯一索引。

### 4.3 `llm.js`
- `generateQuestions`：根據題型與難度生成題目。
- `gradeEssay`：作文/口說輔助評分。
- `generateModelEssay`：範文生成。
- 對外回傳統一資料結構，供 `server.js` 驗證後入庫。

### 4.4 `generate-public.js`
- 從 `templates/public/` 生成 `public/` 靜態頁。
- 目的：讓前端樣板與發布頁一致，降低手改錯誤。

## 5. 核心資料結構
### 5.1 Submission payload（交卷）
```json
{
  "answers": {"<question_id>": "<answer>"},
  "access_code": "optional",
  "submission_id": 123,
  "lookup_token": "hex-token"
}
```

### 5.2 題目安全投影（學生端）
- 包含：題目內容、選項、題型、難度、音訊/圖片、科目資訊。
- 不包含：`answer`、管理端審查欄位。

### 5.3 作答 ownership 欄位
| 欄位 | 說明 |
| --- | --- |
| `student_account_id` | 主要 ownership 主鍵 |
| `student_session_id` | 當次 session 追溯欄位 |
| `lookup_token` | 結果查詢憑證 |

## 6. 關鍵演算法與邏輯
### 6.1 自動評分
- 選擇題/是非題/填充題：標準答案比對。
- `cloze`：以 `|` 分隔逐格比對。
- `writing/speaking`：預設待人工或 AI 輔助評分。

### 6.2 能力估算
- 使用 Rasch 概念計算 `ability` 指標。
- 依題目難度與作答正確性估算個人能力。

### 6.3 推薦策略
- 優先選取錯題相關科目與難度區間。
- 若樣本不足，擴大條件後隨機補足。
- 需排除已作答題或重複內容題。

## 7. 例外與錯誤處理
### 7.1 API 錯誤分類
| 類別 | HTTP 狀態 |
| --- | --- |
| 參數錯誤 | 400 |
| 未認證 | 401 |
| 未授權/越權 | 403 |
| 資源不存在 | 404 |
| 速率限制 | 429 |
| 伺服器錯誤 | 500 |

### 7.2 交易與回滾
- 交卷與明細寫入使用 transaction。
- migration 重建使用 transaction。
- 異常時不保留不一致半成品資料。

## 8. 安全程式規範
1. 所有學生敏感操作必須同時檢查 session 與 ownership。
2. 學生端 API 禁止回傳答案欄位。
3. 上傳 API 必須做 MIME、檔案大小與授權檢查。
4. 密碼只能以雜湊形式存放，不可明文。
5. 登入 API 必須掛載專用 limiter。

## 9. 程式碼品質與審查
### 9.1 Code Review 檢查清單
- 是否破壞既有授權邊界。
- 是否新增未受控的資料讀取路徑。
- 是否更新對應文件與追溯矩陣。
- 是否補齊必要測試或驗證步驟。

### 9.2 靜態檢查建議
- `node --check server.js`
- `node --check database.js`
- `node --check llm.js`

## 10. 需求追溯（程式層）

| FR-011/FR-012 | 作答進度與交卷交易 |
| FR-021 | `/api/public/audio/upload` 授權檢查 |

## 11. 組態與發布
- 程式與文件同版次管理。
- DB 結構變更必須附 migration 與回復說明。
- 發布前需確認 `public/` 與 `templates/public/` 已同步。

## 12. 已知技術債與改善項
| ID | 描述 | 優先級 |
| --- | --- | --- |
| TD-01 | CSP 仍未啟用精細策略 | 中 |
| TD-02 | 缺少完整自動化整合測試 | 中 |
| TD-03 | 安全事件告警仍偏人工 | 中 |

## 13. 驗證與維護要求
1. 每次重大改版需重跑授權回歸測試。
2. 任何 ownership 相關修改必須加測繞過情境。
3. 文件更新需與程式提交同步，不可事後補寫。
4. 發布後需記錄缺陷與改善，回饋至下一輪規格。
