## 2. 專案概述
### 2.1 專案目標
1. 提供穩定且可稽核的線上考試平台，支援題庫管理、出卷、作答、批改、分析與推薦。
2. 以帳號 ownership 綁定學生作答資料，杜絕跨帳號資料讀取與作答次數繞過。
3. 將需求、設計、程式、測試與資安建立可追溯鏈，支援持續稽核與改善。

### 2.2 範圍
- 範圍內：單一站點 Node.js + SQLite 應用、管理端/學生端、AI 產題整合、檔案上傳。
- 範圍外：多租戶隔離、跨區容災、金流與學籍主資料整合。

### 2.3 交付項目
| 類別 | 交付物 |
| --- | --- |
| 管理流程 | 本計畫書、變更單、會議紀錄、風險登錄表 |

- 採用「小團隊快速迭代 + 正式基線控管」模式。
- 裁適原則：不省略需求基線、RTM、CCB、稽核與風險管理。
- 裁適核准：由 PM 與 QA Lead 共同核准並留存記錄。

## 9. 需求管理計畫（REQM）
### 9.1 需求屬性欄位
每一需求至少包含：`ID`、`來源`、`優先級`、`驗證方式`、`狀態`、`版本`、`關聯模組`。

### 9.2 需求狀態流程
`Draft -> Reviewed -> Approved -> Baselined -> Implemented -> Verified -> Validated -> Closed`

### 9.3 變更控制（CCB）
1. 提出 CR（變更理由、影響範圍、風險、回復方案）。
2. SA/DEV/QA 完成影響分析。
3. CCB 會議決議（核准/退回/擱置）。
4. 更新基線與 RTM。
5. 追蹤至驗證完成。

## 10. 組態管理計畫（CM）
### 10.1 組態項目（CI）
- 程式碼：`server.js`、`database.js`、`llm.js`
- 前端：`public/`、`templates/public/`
- 文件：`doc/*.md`
- 資料庫：`exam.db`、migration 腳本與備份檔

### 10.2 基線管理
- 需求基線：SRS v3.x
- 設計基線：SDS/PDS v3.x
- 測試基線：測試案例集與測試報告
- 發布基線：程式 + DB + 文件 + 驗證證據一致版本

### 10.3 版本控制規範
- 每次功能或修補需對應 CR 編號。
- 嚴禁未審查直接變更基線文件。
- 上線前需完成差異比對與發布核准。

## 11. 品質保證計畫（PPQA）
### 11.1 稽核型態
- 流程稽核：檢查是否遵循既定流程與入口/出口準則。
- 產品稽核：檢查工作產品完整性與一致性。

### 11.2 稽核頻率與檢查點
| 稽核項目 | 頻率 | 責任角色 |
| --- | --- | --- |
| 需求/追溯完整性 | 每次基線前 | QA |
| 程式與設計一致性 | 每次 CR 合併前 | DEV Lead/QA |
| 測試證據完整性 | 每輪回歸後 | QA |
| 發布包完整性 | 每次上線前 | PM/OPS |

### 11.3 不符合項處理
- 建立 CAPA（Corrective and Preventive Action）。
- 指派責任人與完成日期。
- 下輪稽核需驗證 CAPA 關閉證據。

## 12. 風險管理計畫（RSKM）
| 風險 ID | 描述 | 影響 | 機率 | 應對策略 | 觸發條件 |
| --- | --- | --- | --- | --- | --- |
| R-01 | 權限控制缺陷導致資料外洩 | 高 | 中 | API 授權檢查 + 回歸測試 | 發現越權案例 |
| R-02 | migration 失敗造成資料不一致 | 高 | 中 | 上線前備份 + 演練回復 | migration 失敗 |
| R-03 | 第三方 LLM 服務不可用 | 中 | 中 | 多供應商 fallback + 關閉 AI 功能 | API 異常率升高 |
| R-04 | 登入暴力嘗試 | 高 | 中 | IP/帳號雙 limiter + 監控警示 | 401 激增 |
| R-05 | 文件與程式脫鉤 | 中 | 中 | 基線審查 + RTM 檢核 | 發現追溯斷點 |

`Issue 登錄 -> 分派 -> 分析 -> 行動 -> 驗證 -> 關閉`

## 16. 驗收與發布管理
### 16.1 上線入口條件
- 高風險缺陷為 0。
- RTM 覆蓋率達成。
- 測試報告與資安報告完成簽核。
- 回復計畫與備份已驗證。

### 16.2 發布核准
- 技術核准：DEV Lead。
- 品質核准：QA Lead。
- 業務核准：PO。
- 最終核准：PM。

### 16.3 上線後監控
- 24 小時內觀察錯誤率、登入失敗率、提交失敗率。
- 發生重大異常時依回復計畫執行。

## 18. 專案結案準則
1. 所有交付物完成並核准。
2. 已知高風險缺陷為 0，剩餘風險完成接受或緩解。
3. 教育訓練與維運移交完成。
4. 專案回顧與改進事項完成登錄。
