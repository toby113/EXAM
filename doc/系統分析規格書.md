## 5. 已定義需求流程（L3）
### 5.1 流程步驟
1. 需求蒐集：訪談、既有系統盤點、法規與資安要求彙整。
2. 需求分析：歸類為功能、非功能、資料、介面與限制需求。
3. 需求規格化：賦予唯一 ID、優先序、來源、驗證方法。
4. 需求審查：SA/DEV/QA/PO 共同審查完整性、可行性、可測試性。
5. 建立基線：核准後凍結版次，列入組態管理。
6. 需求追溯：維護需求到設計、程式、測試與驗收之雙向追溯。
7. 需求變更：透過 CCB 申請、評估、核准後才可更新基線。

### 5.2 需求狀態模型
`Draft -> Analyzed -> Reviewed -> Approved -> Baselined -> Implemented -> Verified -> Validated -> Closed`

### 5.3 進入/退出準則
- 進入準則：需求來源明確、範圍已界定、關鍵利害關係人可參與審查。
- 退出準則：需求已編號、可追溯、可測試、已核准、已納入基線。

## 8. 業務流程分析
### 8.1 登入與授權流程
1. 使用者輸入帳密登入。
2. 系統驗證密碼雜湊（支援舊版自動升級）。
3. 建立 Session Cookie（管理者/學生分流）。
4. 管理端 API 與學生端 API 依 Session 與角色授權。

### 8.2 題庫管理流程
1. 管理者查詢或篩選題目。
2. 新增/編輯題目資料，系統檢核題型、難度與必填欄位。
3. 寫入題庫並更新審查與統計欄位。

### 8.3 試卷管理流程
1. 管理者建立試卷與題目配分。
2. 設定 `draft/active/closed`、時間窗、存取碼、可作答次數。
3. 學生僅可作答符合條件之 `active` 試卷。

### 8.4 學生作答流程
1. 學生登入後建立作答 Session。
2. 系統以 `student_account_id` ownership 建立/續作作答。
3. 作答中定期儲存進度。
4. 交卷後建立 `submissions` 與 `answer_details`，更新題目統計。

### 8.5 人工批改流程
1. 教師查詢待批改明細。
2. 輸入 rubric 分數與評語。
3. 系統回寫明細與總分。

### 8.6 音訊上傳流程（口說題）
1. 前端上傳音訊時提交 `submission_id` 與 `lookup_token`。
2. 後端驗證登入、ownership、token 與作答狀態。
3. 驗證失敗即刪除上傳檔；成功才回傳 `audio_url`。

| --- | --- |

## 14. 驗證與確認策略（VER/VAL）
### 14.1 Verification（建置正確）
- 需求審查：完整性、一致性、可測試性。
- 設計審查：需求對應、介面定義、資料約束一致。
- 程式檢查：lint/語法、關鍵路由權限與資料流程。
- 測試執行：smoke、整合、回歸與資安檢核。

### 14.2 Validation（做對的產品）
- 由 PO/SME 依使用情境執行 UAT：
1. 登入與權限
2. 出題與建卷
3. 學生作答與交卷
4. 批改與分析
5. 異常情境（錯誤 token、越權、超次數）

### 14.3 驗收退出準則
- 高風險缺陷（資安/權限/資料毀損）為 0。
- 主要流程測試通過率 100%。
- 文件、程式、測試結果一致且可追溯。

## 18. 品質保證與稽核（PPQA）
- 每版至少執行一次流程稽核與產品稽核。
- 稽核檢查項：
1. 需求是否有唯一 ID 與驗證方法
2. 需求與程式/測試是否可追溯
3. 高風險缺陷是否關閉
4. migration 是否有備份與回復證據
- 稽核不符合項需建立 CAPA（矯正/預防）並追蹤關閉。

## 19. 決策分析機制（DAR）
### 19.1 需啟用 DAR 的情境
- 資料庫架構重大調整
- 權限模型調整
- 外部 AI Provider 策略調整

### 19.2 決策準則
- 安全性（30%）
- 一致性與可維護性（25%）
- 成本與工期（20%）
- 效能與可擴充性（15%）
- 導入風險（10%）

## 20. 教育訓練與知識移轉（OT）
| 對象 | 訓練主題 | 完成標準 |
| --- | --- | --- |
| DEV | ownership 與資料完整性約束 | 可獨立完成 migration 與回復 |
| QA | 權限/安全測試腳本 | 可覆蓋關鍵越權情境 |
| OPS | 備份/還原與發布流程 | 可完成演練並留存紀錄 |
| SA/PM | CCB 與需求追溯流程 | 可完整維護基線與變更紀錄 |

## 21. 驗收重點
1. 需求、設計、程式、測試可雙向追溯。
2. 學生端無法取得題目正解與他人作答資料。
3. 作答 ownership 與資料庫完整性約束生效。
4. 登入與上傳行為具防濫用控制。
5. 變更有紀錄、可審計、可回復。
